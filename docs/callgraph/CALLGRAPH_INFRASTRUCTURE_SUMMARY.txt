💾 OUTPUT FORMATS

1. DOT FORMAT (tool-callgraph.dot)
   ├─ Graphviz compatible (apt install graphviz)
   ├─ Visual encoding:
   │  ├─ Boxes (yellow) = User functions
   │  ├─ Boxes (green) = Entry point
   │  ├─ Ellipses (blue) = Library APIs
   │  ├─ Ellipses (gray) = CLI-only
   │  ├─ Diamonds (orange) = AST gates
   │  ├─ Red edges (thick) = NULL-guarded calls
   │  └─ Dashed edges = Gate branches
   └─ Rendering:
      ├─ dot -Tpng graph.dot -o graph.png
      ├─ dot -Tsvg graph.dot -o graph.svg
      └─ dot -Tpdf graph.dot -o graph.pdf

   ├─ Top-level: source, functions[], call_sites[], ast_gates[]
   ├─ Includes: summary, fidelity metrics
   ├─ Functions: name, lines, params, return_type, call_count
   ├─ CallSites: caller, callee, line, gate, cli_only
   ├─ Gates: condition, line, type, parent, true_calls[], false_calls[]
   └─ Fidelity: matched, fuzzable, percentage, detailed lists

3. SVG/PNG (tool-callgraph.svg, .png)
   ├─ Generated by dot command (requires graphviz)
   └─ Human-readable visualization

4. TREE FORMAT (PrintCallChainTree() - IccAnalyzerCallGraph.cpp:325)
   ├─ ASCII art with indentation
   ├─ Shows entry → crash path

   └─ Exploitability verdict


🔍 REGEX PATTERNS (C++ parser, IccAnalyzerCallGraph.cpp)
───────────────────────────────────────────────────────────────────────────────

ASAN Error Detection:
  "ERROR: AddressSanitizer: ([^\s]+)"
  Captures: stack-buffer-overflow, heap-buffer-overflow, use-after-free, etc.

Access Info:
  "(READ|WRITE) of size (\d+)"
  Captures: access_type, access_size

Memory Location:
  "on address (0x[0-9a-f]+)"
  Captures: crash address

Stack Variable:
  "\[(\d+), (\d+)\) '([^']+)'.*<== Memory access at offset (\d+)"
  Captures: var_start, var_end, var_name, overflow_offset

Stack Frame:
  "\s*#(\d+)\s+0x[0-9a-f]+\s+in\s+(.+?)\s+([^\s:]+\.(?:cpp|c|cc|cxx|h|hpp)):(\d+)"
  Captures: frame_num, function_name, file_path, line_number

UBSAN Runtime Error:
  "([^:\s]+):(\d+):(\d+): runtime error: (.+)"
  Captures: file, line, column, error_message

⚠️ LIMITATIONS & DESIGN NOTES
───────────────────────────────────────────────────────────────────────────────

✓ Strengths:
  • Hand-verified accuracy (not regex-based parsing)
  • Comprehensive gate tracking with security annotations
  • Multi-format output (JSON, DOT, SVG, PNG, tree)
  • Real crash mapping (ASAN/UBSAN integration)
  • Quantified fidelity metrics (90% aggregate)
  • Security-focused (NULL guards, validation paths)

✗ Limitations:
  • Manual updates required when source changes
  • No incremental re-analysis
  • Indirect calls tracked but not fully utilized (is_indirect unused)
  • Python scripts tightly coupled to specific tools
  • No cross-tool dependency graph
  • No automated fuzzer alignment validation
  • Limited API contract consistency checking
