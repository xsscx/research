[P0.1] BUFFER OVERFLOW in IccHeuristicsRawPost.cpp
────────────────────────────────────────────────────
File:Line: IccHeuristicsRawPost.cpp:281, 292, 308
Pattern: Repeated buffer overflow in H35 heuristic

Issue:
  Multiple heuristics (H33, H34, H35) use fixed 256-byte buffer for binary data
  but don't validate the actual data size before reading.
  
  Line 281: icUInt8Number bData[256];
  Line 287-289: Loop reads bData[b] with dataLen from untrusted file
  
  If (dataLen > 256), buffer overflows when accessed via line 289:
    if (bData[b] == bData[b-1])  // OOB read if b >= 256
  
  Also duplicated code: sig35[5] (lines 292, 308) filled with 4-byte casts
  without null termination in some paths.

Impact: Stack buffer overflow → crash or RCE on malicious profiles

Fix:
  Line 281: Change to dynamic allocation:
    size_t bDataLen = (dataLen > 256) ? 256 : dataLen;
    std::vector<uint8_t> bData(bDataLen);
  
  Or add bounds check before loop (line 287):
    for (size_t b = 1; b < bDataLen && b < 256; b++) {

[P0.2] UNCHECKED malloc() RETURN in IccHeuristicsProfileCompliance.cpp
────────────────────────────────────────────────────
File:Line: IccHeuristicsProfileCompliance.cpp:619, 625
Pattern: Null pointer dereference after malloc

Code:
  619:  unsigned char *buf = (unsigned char *)malloc(scanSize);
  620:  if (!buf) { printf("\n"); return 0; }
  
ISSUE: malloc() can fail, but code at line 625 proceeds to use buf:
  625:  size_t bytesRead = fread(buf, 1, scanSize, fh.fp);
  
While there IS a null check at line 620, the control flow is confusing.
However, further scanning: line 625+ accesses buf[i] in loop at line 628:
  for (size_t i = 128; i + 16 <= bytesRead; ) {
    if (buf[i] == 0x90) {  // Dereference after successful malloc
    
ACTUAL PROBLEM: No validation of fread return value before using bytesRead.
If fread fails (I/O error), bytesRead is uninitialized or garbage.

Fix:
  625: size_t bytesRead = fread(buf, 1, scanSize, fh.fp);
  Add: if (bytesRead < 128) { free(buf); printf("\n"); return 0; }

[P1.1] COMMAND INJECTION VIA snprintf() in IccAnalyzerSecurity.cpp
────────────────────────────────────────────────────
File:Line: IccAnalyzerSecurity.cpp:69-85
Function: static std::string RunExternalTool(...)

Code:
  74:  char cmd[4096];
  75:  int n = snprintf(cmd, sizeof(cmd), "timeout 10 %s %s '%s' 2>&1",
  76:                   tool, args, filename);
  77:  if (n < 0 || static_cast<size_t>(n) >= sizeof(cmd)) return "";

ISSUE: Caller-supplied 'args' parameter is NOT validated for shell metacharacters.
While 'filename' is validated (line 72: strpbrk check), 'args' is directly
inserted into the command string.

Example attack:
  RunExternalTool("exiftool", "; rm -rf /", "/tmp/file.icc")
  → Executes: "timeout 10 exiftool ; rm -rf / '/tmp/file.icc' 2>&1"

Impact: Remote Code Execution if 'args' comes from untrusted input

Fix (Line 72-73):
  Before the snprintf, add:
    if (strpbrk(args, ";|&$`\\\"'{}()!<>") != nullptr) {
      return "";
    }

[P2.7] INCONSISTENT BUFFER MANAGEMENT in IccAnalyzerJson.cpp
────────────────────────────────────────────────────
File:Line: IccAnalyzerJson.cpp:50-65 (StripAnsi function)
Pattern: Manual string concatenation without bounds checking

Code:
  while (i < s.size()) {
    if (s[i] == '\033' && i + 1 < s.size() && s[i + 1] == '[') {
      i += 2;
      while (i < s.size() && s[i] != 'm') i++;
      if (i < s.size()) i++; // skip 'm'

Issue:
  String building via out += uses std::string reserve (good), but logic
  advances i without guaranteed bounds after inner loops.
  
  If malformed ANSI sequence: "\\033[" without 'm', loop exits with i unchanged,
  falls through to else clause adding raw char.

Fix (Line 58):
  while (i < s.size()) {
    if (s[i] == '\033' && i + 1 < s.size() && s[i + 1] == '[') {
      i += 2;
      while (i < s.size() && s[i] != 'm') i++;
      if (i < s.size() && s[i] == 'm') i++;  // Only advance if 'm' found
      continue;  // Don't add the ESC char
    }
    out += s[i++];
  }
